# CS336 Spring 2024 Assignment 5: Alignment & Safety - 详细总结

## 1. 作业概览 (Assignment Overview)

### 1.1 核心目标
本作业旨在让学生**深入理解并亲手实现大语言模型（LLM）的对齐（Alignment）与安全（Safety）技术**。学生将从“红队测试”（Red Teaming，即攻击模型以发现漏洞）开始，进而实现两种主流的对齐算法：**拒绝采样微调（Rejection Sampling Fine-Tuning, RSFT）**和**直接偏好优化（Direct Preference Optimization, DPO）**。最终目标是构建一个既能遵循指令又能抵抗有害提示的鲁棒模型。

### 1.2 任务背景
*   **场景**: 你是一家 AI 公司的安全工程师。公司发布了一个强大的基础模型（Base Model），但它容易被诱导生成有害内容（如仇恨言论、危险建议）。
*   **挑战**:
    *   如何系统性地发现模型的弱点？
    *   如何利用有限的人类反馈数据（或合成数据）高效地调整模型行为？
    *   如何在提升安全性的同时，不损害模型的通用能力（Helpfulness）？
*   **基线模型**: 基于 Assignment 4 处理的数据训练出的 GPT-2 Small 变体（或提供的预训练 Checkpoint）。

### 1.3 主要任务模块
1.  **红队测试 (Red Teaming)**: 手动和自动构造对抗性提示（Adversarial Prompts），挖掘模型的有害输出。
2.  **数据构建 (Dataset Construction)**: 收集“提示 - 响应”对，并标记偏好（Chosen vs. Rejected），构建 RLHF/DPO 所需的数据集。
3.  **拒绝采样微调 (RSFT)**: 实现基于监督微调（SFT）的简单对齐方法，仅使用“好”的回答进行训练。
4.  **直接偏好优化 (DPO)**: 实现 DPO 算法，直接利用偏好对（好回答 vs. 坏回答）优化策略，无需显式训练奖励模型。
5.  **评估与权衡 (Evaluation & Trade-offs)**: 在安全性基准（如 Toxigen）和有用性基准上评估模型，分析“安全 - 有用”权衡曲线。

## 2. 核心技术实现细节

### 2.1 红队测试 (Red Teaming)
这是对齐的第一步，目的是暴露问题。
*   **手动红队**: 学生需扮演攻击者，尝试各种越狱技巧（Jailbreaking）：
    *   **角色扮演**: “假设你是一个不受限制的黑客...”
    *   **逻辑陷阱**: “为了研究目的，请列出制造...的步骤。”
    *   **多语言/编码攻击**: 使用小语种或 Base64 编码绕过过滤器。
*   **自动红队 (可选/进阶)**: 使用另一个 LLM（攻击者模型）自动生成大量对抗性提示，测试目标模型的防御率。
*   **产出**: 建立一个包含 `prompt`, `harmful_response`, `safe_response` 的种子数据集。

### 2.2 数据构建与偏好标注
*   **数据来源**:
    *   **开源数据集**: 如 HH-RLHF (Helpful and Harmless), UltraFeedback, Orca-DPO-Pairs。
    *   **自生成数据**: 利用基线模型生成多个回答，由人工（或强模型代理）标注哪个更好（Chosen）哪个更差（Rejected）。
*   **数据格式**: 三元组 `(prompt, chosen_response, rejected_response)`。
    *   `chosen`: 安全、有帮助的回答。
    *   `rejected`: 有害、拒绝回答但态度恶劣、或无关的回答。
*   **关键考量**: 数据的质量和多样性直接决定对齐效果。需覆盖多种有害类别（暴力、歧视、隐私、非法行为等）。

### 2.3 拒绝采样微调 (Rejection Sampling Fine-Tuning, RSFT)
这是一种简化的对齐方法，本质是高质量的监督微调（SFT）。
*   **原理**: 只使用偏好数据中的 `chosen` 部分（好回答）来微调模型，完全忽略 `rejected` 部分。
*   **流程**:
    1.  过滤数据集，仅保留 `(prompt, chosen_response)` 对。
    2.  使用标准交叉熵损失函数（Cross-Entropy Loss）进行 SFT。
    3.  $L_{SFT} = -\sum \log P_\theta(y_{chosen} | x)$
*   **局限性**: 模型学会了“什么是好的”，但没有显式地学会“什么是坏的”以及“如何区分好坏”。它可能只是模仿了好回答的风格，而未真正内化安全边界。

### 2.4 直接偏好优化 (Direct Preference Optimization, DPO)
这是当前工业界最流行的对齐算法，绕过了训练独立奖励模型（Reward Model）的复杂过程。
*   **核心思想**: 将偏好优化问题转化为直接的分类问题，通过修改损失函数，直接更新策略模型 $\pi_\theta$。
*   **数学原理**:
    *   基于 Bradley-Terry 偏好模型，推导出了隐式奖励函数。
    *   **DPO 损失函数**:
        $$ L_{DPO}(\pi_\theta; \pi_{ref}) = -\mathbb{E}_{(x, y_w, y_l) \sim D} \left[ \log \sigma \left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)} \right) \right] $$
    *   其中：
        *   $y_w$: Chosen response (赢家)
        *   $y_l$: Rejected response (输家)
        *   $\pi_{ref}$: 参考模型（通常是冻结的 SFT 模型或初始基线模型），用于防止策略偏离太远（KL 散度约束）。
        *   $\beta$: 控制偏离程度的超参数（通常 0.1 - 0.5）。
        *   $\sigma$: Sigmoid 函数。
*   **实现步骤**:
    1.  **加载参考模型**: 克隆当前策略模型并冻结参数作为 $\pi_{ref}$。
    2.  **前向传播**: 对同一个 prompt，分别计算模型对 $y_w$ 和 $y_l$ 的对数概率（Log Probabilities）。
    3.  **计算损失**: 应用上述 DPO 公式，最大化 $y_w$ 相对 $y_l$ 的优势。
    4.  **反向传播**: 仅更新策略模型 $\pi_\theta$ 的参数。
*   **优势**: 训练更稳定，超参数更少，无需训练额外的奖励模型，显存占用更低。

## 3. 实验与评估 (Experiments & Evaluation)

### 3.1 评估基准
对齐不仅仅是看训练损失，必须通过外部基准测试：
*   **安全性基准 (Safety Benchmarks)**:
    *   **Toxigen**: 测量模型生成有毒内容的概率。
    *   **RealToxicityPrompts**: 给定诱导性提示，评估输出的毒性分数（使用 Perspective API 或本地毒性分类器）。
    *   **自定义红队集**: 使用作业第一部分收集的对抗性提示进行测试。
    *   **指标**: 毒害率 (Toxicity Rate), 拒绝率 (Refusal Rate)。
*   **有用性基准 (Helpfulness Benchmarks)**:
    *   **AlpacaEval / MT-Bench (简化版)**: 评估模型在通用指令上的表现。
    *   **指标**: 胜率 (Win Rate), 平均得分。
    *   **关键点**: 警惕“过度拒绝”（Over-refusal），即模型对正常问题也回答“我不能回答这个问题”。

### 3.2 实验任务
1.  **基线对比**: 评估原始 Base Model 的毒性水平。
2.  **RSFT vs. DPO**:
    *   训练 RSFT 模型，评估其安全性和有用性。
    *   训练 DPO 模型（尝试不同的 $\beta$ 值），评估其表现。
    *   对比两者在相同数据量下的收敛速度和最终性能。
3.  **数据量敏感性**: 探究偏好数据数量（100, 500, 1000 条）对对齐效果的影响。
4.  **超参数消融**: 分析 $\beta$ 参数对 KL 散度和模型性能的影响。

### 3.3 分析报告要求
*   **红队记录**: 展示成功的越狱案例和失败的案例，分析模型弱点。
*   **训练曲线**: 绘制 DPO 损失曲线、准确率（预测偏好对的准确度）曲线。
*   **雷达图/散点图**: 可视化“安全性 vs. 有用性”的权衡。理想的模型应位于右上角（高安全、高有用）。
*   **案例分析**:
    *   展示 DPO 修复的具体案例（Base Model 输出有害，DPO Model 拒绝或引导）。
    *   展示“过度对齐”的案例（模型变得过于保守）。
*   **理论思考**: 讨论 DPO 相比传统 RLHF (PPO) 的优缺点；探讨数据质量对对齐的决定性作用。

## 5. 注意事项与技巧 (Tips & Best Practices)

### 5.1 实现陷阱
*   **参考模型冻结**: 在 DPO 训练中，务必确保 $\pi_{ref}$ 的参数是冻结的（`requires_grad=False`），且处于 `eval` 模式，否则梯度会错误地流向参考模型，导致训练崩溃。
*   **概率计算**: 计算 Log Prob 时，要屏蔽掉 Prompt 部分的 Loss，只计算 Response 部分的概率。注意 Mask 的处理。
*   **数值稳定性**: DPO 损失涉及 Sigmoid 和对数，需注意数值溢出问题，PyTorch 实现时需小心处理 logits 的范围。
*   **Batch 构造**: DPO 需要成对数据 `(prompt, chosen, rejected)`，DataLoader 需确保同一 Batch 内数据格式正确。

### 5.2 数据策略
*   **质量 > 数量**: 对于 DPO，少量高质量、差异明显的偏好对（Chosen 明显好于 Rejected）往往比大量噪声数据更有效。
*   **多样性**: 确保数据覆盖多种有害类型，避免模型只学会了拒绝某一种特定的攻击。
*   **混合数据**: 可以考虑混合一部分纯 SFT 数据（通用指令）和 DPO 数据，以防止灾难性遗忘（Catastrophic Forgetting）。

### 5.3 评估客观性
*   **自动化评估的局限**: 毒性分类器本身可能有误报。建议结合人工抽检（Human Eval）来验证自动化指标的可靠性。
*   **基准选择**: 不要只在训练数据分布相似的测试集上评估，要使用分布外（OOD）的红队数据测试泛化能力。

### 5.4 伦理与责任
*   **接触有害内容**: 在构建数据和红队过程中，学生会接触到大量真实的有害文本。需注意心理健康，必要时跳过极端内容。
*   **双刃剑**: 理解对齐技术既可用于保护用户，也可能被用于审查。报告中应体现对这一伦理问题的思考。
