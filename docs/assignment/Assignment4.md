# CS336 Spring 2024 Assignment 4: Filtering Language Modeling Data - 详细总结

## 1. 作业概览 (Assignment Overview)

### 1.1 核心目标
本作业旨在让学生**亲手构建从原始网页爬虫数据到高质量语言模型训练数据集的完整流水线**。学生将模拟工业界数据工程师的角色，处理 Common Crawl 原始数据，通过一系列过滤、清洗和去重步骤，最终训练出一个在特定基准测试上表现优异的 Transformer 语言模型。

### 1.2 任务背景
*   **数据来源**: Common Crawl (CC)，一个包含超过 2500 亿网页的非营利性公开网络爬虫数据集。
*   **挑战**: 原始 CC 数据包含大量噪声（HTML 标签、低质量内容、重复数据、有害信息、隐私泄露等），直接用于训练会导致模型性能下降或产生有害输出。
*   **最终目标**: 构建一个数据处理流水线，过滤 5000 个 WARC 文件（约 6TB 压缩数据），训练一个 GPT-2 Small 模型，并在 **C4 100 domains (Paloma Benchmark)** 验证集上获得最低的困惑度 (Perplexity)。

### 1.3 主要任务模块
1.  **数据探索与理解**: 分析 WARC/WET 文件格式，理解原始数据的噪声特性。
2.  **文本提取 (HTML to Text)**: 使用 `Resiliparse` 库从 HTML 中提取纯文本，处理编码问题。
3.  **内容过滤 (Filtering)**:
    *   **语言识别**: 使用 `fastText` 筛选特定语言（如英语）。
    *   **隐私保护 (PII Masking)**: 正则匹配并掩码邮箱、电话、IP 地址。
    *   **有害内容检测**: 使用 `fastText` 分类器识别 NSFW 和有毒言论。
    *   **启发式质量规则**: 基于 Gopher 论文的规则（长度、符号比等）过滤低质文档。
    *   **学习式质量分类**: 训练自定义分类器区分高/低质量文本。
4.  **去重 (Deduplication)**:
    *   **精确行去重**: 移除 corpus 中重复出现的行。
    *   **模糊文档去重**: 使用 MinHash + LSH 算法移除近似重复的文档。
5.  **大规模并行处理**: 使用 `concurrent.futures` 或 `submitit` 在集群上并行处理海量数据。
6.  **模型训练与评估**: 将处理后的数据 Tokenize，训练模型并提交排行榜结果。

## 2. 核心技术实现细节

### 2.1 数据格式与探索 (Data Formats & Exploration)
*   **WARC (Web ARChive)**: 包含原始 HTTP 响应、元数据（URL, 时间戳）和 HTML 内容。是处理的起点。
*   **WET (Web Extracted Text)**: Common Crawl 提供的预提取文本，但通常包含大量噪声（如菜单、页脚），需重新评估或替换。
*   **任务**: 手动检查样本，识别典型噪声（导航栏、广告、乱码），理解为何需要自定义过滤 pipeline。

### 2.2 文本提取 (HTML to Text Conversion)
*   **工具**: `Resiliparse` 库 (`extract_plain_text`)。
*   **关键步骤**:
    *   **编码检测**: 网页编码不全是 UTF-8，需使用 `resiliparse.parse.encoding.detect_encoding` 自动检测并解码。
    *   **提取逻辑**: 解析 HTML 树，提取可见文本块，去除脚本、样式等不可见内容。
*   **对比**: 比较自提取文本与官方 WET 文件的差异，分析哪种更适合训练。

### 2.3 内容过滤策略 (Content Filtering Strategies)

#### 2.3.1 语言识别 (Language Identification)
*   **工具**: `fastText` 预训练模型 (`lid.176.bin`)。
*   **实现**: 输入文本 -> 预测语言标签 + 置信度分数。
*   **策略**: 设定置信度阈值（如 >0.8），仅保留目标语言（通常是英语）文档。
*   **风险**: 误判多语言混合文档；需人工抽检评估错误率。

#### 2.3.2 隐私信息掩码 (PII Masking)
*   **目标**: 防止模型记忆并泄露个人隐私。
*   **实现**: 使用正则表达式 (Regex) 匹配并替换：
    *   **Email**: `\| \| \| EMAIL_ADDRESS\| \| \|`
    *   **Phone**: 覆盖常见美国格式及变体 -> `\| \| \| PHONE_NUMBER\| \| \|`
    *   **IP (IPv4)**: `xxx.xxx.xxx.xxx` 格式 -> `\| \| \| IP_ADDRESS\| \| \|`
*   **副作用**: 可能误杀非隐私数字（如年份、版本号），需在报告中讨论。

#### 2.3.3 有害内容检测 (Harmful Content Detection)
*   **工具**: Dolma 项目提供的 `fastText` 分类器（基于 Jigsaw Toxic Comments 数据集训练）。
    *   **NSFW 分类器**: 识别色情、亵渎等内容。
    *   **Toxic Speech 分类器**: 识别仇恨言论、攻击性语言。
*   **实现**: 输入文本 -> 返回标签 (Toxic/Non-Toxic) + 置信度。
*   **策略**: 设定阈值过滤高分有害文档；需平衡召回率与误杀率（避免过度审查）。

#### 2.3.4 启发式质量规则 (Heuristic Quality Rules - Gopher)
参考 Gopher 论文 [Rae et al., 2021]，实施以下规则过滤低质文档：
*   **长度限制**: 单词数 < 50 或 > 100,000。
*   **平均词长**: 不在 3-10 字符之间。
*   **省略号比例**: > 30% 的行以 "..." 结尾（通常表示截断或列表）。
*   **字母词比例**: < 80% 的单词包含至少一个字母（过滤代码、乱码）。

#### 2.3.5 学习式质量分类器 (Learned Quality Classifier)
*   **动机**: 启发式规则无法捕捉语义质量。
*   **数据构建**:
    *   **正样本**: 从 Wikipedia 引用链接中爬取的高信誉页面（需先经过基础过滤）。
    *   **负样本**: Common Crawl 中的随机页面。
*   **模型**: 训练一个 `fastText` 分类器，输出质量分数。
*   **应用**: 根据分数排序或设定阈值，保留高质量文档。

### 2.4 去重技术 (Deduplication Techniques)

#### 2.4.1 精确行去重 (Exact Line Deduplication)
*   **原理**: 统计整个语料库中每行出现的频率，仅保留出现次数为 1 的行。
*   **优化**: 使用行的 Hash 值作为 Key 存储计数器，节省内存。
*   **效果**: 有效移除模板化内容（如网站页眉、页脚、导航栏）。

#### 2.4.2 模糊文档去重 (Fuzzy Document Deduplication)
*   **算法**: **MinHash + LSH (Locality Sensitive Hashing)**。
*   **流程**:
    1.  **Shingling**: 将文档转换为 n-gram 集合。
    2.  **MinHash**: 生成签名向量 (Signature)，近似估计 Jaccard 相似度。
    3.  **LSH 分桶**: 将签名分为 $b$ 个 band，每个 band $r$ 行。若两个文档在任一 band 签名完全相同，则视为候选重复对。
    4.  **验证**: 对候选对计算真实的 Jaccard 相似度，超过阈值则判定为重复。
    5.  **聚类移除**: 将重复文档聚类，每类随机保留一个。
*   **参数调优**: 调整 Hash 数量、Band 数量、n-gram 大小以平衡精度与召回率。
*   **预处理**: 计算前需对文本进行归一化（小写、去标点、去重空格、Unicode 规范化）以提高鲁棒性。

### 2.5 大规模并行处理 (Scalable Processing)
*   **挑战**: 处理 5000+ WARC 文件（6TB+）耗时巨大。
*   **方案**:
    *   **单机多核**: 使用 Python `concurrent.futures.ProcessPoolExecutor`。
    *   **集群调度**: 使用 `submitit` 库提交 Slurm 作业数组 (Job Array)，配置 CPU/内存资源，自动管理任务队列和结果收集。
*   **代码结构**: 定义 `process_single_warc_file` 函数，并行映射到所有输入文件。

## 3. 实验与排行榜挑战 (Experiments & Leaderboard)

### 3.1 数据过滤流水线设计
*   **自由组合**: 学生可自由选择上述过滤原语的组合顺序和参数（如阈值、n-gram 大小）。
*   **迭代优化**:
    1.  运行初步过滤，统计各阶段丢弃比例。
    2.  抽样检查被丢弃和保留的文档，人工评估质量。
    3.  调整参数或增加新规则（如基于 n-gram LM 困惑度的过滤）。
    4.  重新运行并观察变化。

### 3.2 数据量化与 Tokenization
*   **统计**: 报告最终保留的文档数、Token 总数。
*   **Tokenizer**: 使用 GPT-2 Tokenizer (`tiktoken` 或 `transformers`)。
*   **序列化**: 将 Token ID 序列保存为 `np.uint16` 二进制文件，适配训练脚本。
*   **特殊 Token**: 每个文档后添加 `<|endoftext|>`。

### 3.3 模型训练与评估
*   **模型架构**: GPT-2 Small (12 layers, 768 hidden, 12 heads, ~124M params)。**严禁修改模型结构或超参数**。
*   **训练配置**:
    *   步数: 200K iterations。
    *   Batch Size: 128 per GPU (2 GPUs)。
    *   Optimizer: AdamW, LR 1e-3 (Cosine decay), Warmup 1%。
    *   Precision: BF16。
*   **评估指标**: **C4 100 domains (Paloma Benchmark)** 验证集上的困惑度 (Perplexity) 或 Loss。
*   **排行榜**: 提交最佳验证 Loss 至 GitHub 排行榜。

### 3.4 分析报告要求
*   **过滤统计**: 详细列出每一步过滤器丢弃的数据比例。
*   **案例分析**:
    *   展示 5 个保留的高质量样本及其理由。
    *   展示 5 个被丢弃/修改的样本，分析过滤是否合理（False Positives/Negatives）。
*   **迭代记录**: 描述尝试过的不同配置及其对最终结果的影响。
*   **时间成本**: 估算处理全部 Common Crawl (90k 文件) 所需的时间。

## 5. 注意事项与技巧 (Tips & Best Practices)

### 5.1 开发策略
*   **小规模调试**: 先在 1-5 个 WARC 文件上调试代码，确保逻辑正确后再扩展到 5000 个。
*   **缓存中间结果**: 过滤过程耗时，建议分阶段保存中间结果（如提取文本后、去重后），避免重复计算。
*   **监控资源**: 使用 `submitit` 时合理申请 CPU 和内存，避免 OOM 或超时。

### 5.2 过滤平衡
*   **过犹不及**: 过度过滤可能导致数据量不足，模型欠拟合；过滤不足则引入噪声。需在“数据量”与“数据质量”间寻找平衡点。
*   **阈值敏感性**: 对分类器阈值进行敏感性分析，观察不同阈值下保留数据的质量与数量变化。

### 5.3 去重陷阱
*   **MinHash 参数**: Band 数量和 Hash 数量直接影响召回率和精度。需根据数据规模和经验公式（如 $b \approx t^{-r}$）调整。
*   **归一化**: 模糊去重前的文本归一化至关重要，否则微小的格式差异会导致漏检。

### 5.4 伦理与安全
*   **有害内容**: 处理过程中可能接触到极端有害内容，需注意心理防护，必要时跳过。
*   **隐私合规**: 即使掩码了 PII，也需意识到完全匿名化的难度，报告中应讨论潜在风险。

> **注**: 本作业是连接数据工程与大模型训练的实战演练。成功的关键不仅在于代码实现，更在于对数据质量的敏锐直觉和科学的实验迭代方法。通过构建高效的过滤流水线，学生将深刻理解“Data-Centric AI”的核心理念。
